# PROJECT RULE — Portfolio

Este é o projeto ativo principal do portfólio profissional de Henrico Amaral.

Este projeto possui escopo próprio, memória própria e Git próprio.

Nunca tratar Portfolio como parte genérica da pasta !PROJETOS.

Todos os commits, alterações, testes, builds e decisões deste projeto devem acontecer somente dentro da pasta:

/Portfolio

## OBJETIVO DO PROJETO

Criar e evoluir o portfólio profissional atual de Henrico Amaral.

O objetivo não é apenas mostrar telas bonitas.

O objetivo é reposicionar Henrico como Senior Product Designer especializado em:

- sistemas complexos
- plataformas enterprise
- arquitetura de informação
- visualização operacional
- design systems
- produtos digitais de alta complexidade
- interfaces para decisão
- organização de fluxos críticos
- tradução de operação real em clareza

O portfólio deve funcionar como uma peça estratégica de reposicionamento profissional.

## POSICIONAMENTO

Henrico não deve ser apresentado como designer visual genérico.

Ele deve ser apresentado como um designer sênior capaz de estruturar complexidade real, organizar problemas difíceis e transformar sistemas ambíguos em interfaces claras para decisão.

O portfólio deve comunicar:

- maturidade profissional
- pensamento sistêmico
- clareza estratégica
- domínio de produto
- capacidade de análise
- experiência com ambientes complexos
- qualidade visual sóbria
- precisão narrativa
- domínio técnico de interface
- capacidade de criar sistemas, não apenas telas

## REGRAS DE ESCOPO

Ao trabalhar em Portfolio, nunca alterar:

- ../LavaPro
- ../SaaS
- ../OLD_portfolio
- ../!SKILLS
- arquivos da raiz !PROJETOS

salvo pedido explícito.

Não mover arquivos entre projetos.

Não usar decisões de outro projeto como se fossem decisões deste portfólio.

## REGRAS DE DESIGN

O portfólio deve ter estética:

- técnica
- editorial
- madura
- sóbria
- precisa
- sofisticada
- orientada a sistemas
- sem aparência de template genérico

Evitar:

- visual genérico de SaaS
- landing page comum
- excesso de glow
- excesso de gradiente
- glassmorphism gratuito
- grid forte demais
- motion chamativo
- texto motivacional
- copy genérica de UX
- cards decorativos sem função
- seções visualmente desconectadas

Preferir:

- hierarquia visual clara
- composição editorial
- backgrounds sutis
- motion sofisticado e discreto
- tipografia forte e legível
- espaçamento consistente
- responsividade real
- densidade visual controlada
- linguagem textual concreta
- narrativa baseada em complexidade real

## REGRAS DE CONTEÚDO

Evitar frases genéricas como:

- experiências incríveis
- soluções inovadoras
- design centrado no usuário
- interfaces intuitivas
- jornada fluida
- paixão por resolver problemas
- transformação digital

Preferir linguagem concreta:

- sistemas de decisão
- arquitetura de informação
- visualização operacional
- coordenação de fluxos
- redução de ambiguidade
- leitura de cenários complexos
- infraestrutura de produto
- clareza para operação
- design para sistemas críticos

## REGRAS TÉCNICAS

Antes de alterar código:

1. Identificar o arquivo certo.
2. Entender o padrão existente.
3. Evitar refatoração desnecessária.
4. Evitar instalar dependências sem motivo.
5. Preservar build.
6. Preservar responsividade.
7. Preservar acessibilidade básica.
8. Fazer alteração incremental.
9. Atualizar memória se houver decisão relevante.

Não reescrever componentes inteiros sem necessidade.

Não duplicar lógica.

Não criar estilos isolados que quebrem o sistema visual.

## REGRAS DE GIT

Commits deste projeto devem acontecer apenas dentro de:

/Portfolio

Antes de commit:

1. Confirmar que o terminal está em /Portfolio.
2. Rodar git status.
3. Verificar se só existem arquivos deste projeto modificados.
4. Confirmar que a memória local foi atualizada, se necessário.
5. Usar mensagem de commit clara.

Formato recomendado:

type(scope): resumo curto

Exemplos:

system(memory): add persistent project memory
design(home): refine hero section hierarchy
content(cases): improve case cards copy
fix(layout): correct mobile spacing
docs(memory): update visual decisions

## REGRA FINAL

Nunca trabalhar como se este portfólio estivesse começando do zero.

Sempre consultar a memória, preservar decisões consolidadas, proteger a coerência do sistema e atualizar o registro quando houver evolução relevante.
