# DESIGN SYSTEM MEMORY — Portfolio

Este arquivo registra a memória do sistema visual do portfólio Portfolio.

Ele deve ser atualizado sempre que uma decisão visual sistêmica for consolidada.

## Status

Sistema visual em evolução ativa.

Não considerar decisões visuais antigas como definitivas sem verificar o código atual.

Não reutilizar paletas, estilos ou direções antigas de outros projetos sem validação explícita.

## Direção visual

O portfólio deve ter estética:

- técnica
- editorial
- madura
- sóbria
- precisa
- sofisticada
- autoral
- orientada a sistemas
- próxima de produtos complexos e plataformas enterprise

A interface deve parecer uma peça de posicionamento profissional sênior, não uma landing page genérica.

## Princípios visuais

### 1. Clareza antes de decoração

Todo elemento visual precisa ter função.

Evitar elementos decorativos que não reforcem narrativa, hierarquia ou profundidade.

### 2. Sistema antes de seção isolada

Nenhuma seção deve parecer criada fora do sistema.

Hero, Método, Cases e CTA devem compartilhar coerência de:

- background
- tipografia
- espaçamento
- densidade
- motion
- bordas
- grid
- contraste
- ritmo visual

### 3. Backgrounds sutis

Backgrounds podem ter camadas técnicas, grids, texturas e motion, mas devem ser sutis.

O conteúdo sempre deve ser mais importante que o fundo.

### 4. Grid controlado

Grid deve sugerir estrutura e precisão.

Evitar grid forte demais, que pareça interface futurista genérica ou que concorra com o conteúdo.

### 5. Motion funcional

Motion deve reforçar continuidade, leitura e profundidade.

Evitar animações chamativas.

### 6. Densidade controlada

O portfólio pode ter densidade técnica, mas não pode parecer poluído.

A leitura precisa continuar clara.

## Tipografia

Regras gerais:

- títulos devem ter presença
- texto de apoio deve ser legível
- evitar fonte pequena demais
- manter hierarquia clara
- evitar excesso de pesos
- preservar leitura em mobile

A tipografia deve comunicar maturidade, não fragilidade.

## Espaçamento

Usar espaçamento como estrutura.

Preferir escala consistente baseada em múltiplos de 4px ou 8px.

Evitar:

- seções apertadas
- respiros aleatórios
- cards desalinhados
- blocos visuais com densidade incoerente

## Cards

Cards devem ter função narrativa ou estrutural.

Evitar cards apenas decorativos.

Cards de cases devem comunicar:

- tipo de problema
- contexto
- complexidade
- papel do designer
- força visual do case
- conexão com o posicionamento geral

## Imagens

As imagens dos cases devem parecer:

- cinematográficas
- técnicas
- sofisticadas
- relacionadas ao problema real
- não genéricas
- não decorativas
- coerentes com a narrativa do case

As imagens devem reforçar a percepção de complexidade e sistema.

## Motion

Motion aprovado deve ser:

- lento
- sutil
- elegante
- contextual
- quase imperceptível nos backgrounds
- funcional em entradas de conteúdo

Evitar:

- loops chamativos
- parallax exagerado
- elementos pulando
- scale agressivo
- blur excessivo
- animações que dificultem leitura

## Responsividade

Toda seção deve funcionar bem em:

- desktop
- notebook
- tablet
- mobile

Não aceitar layout que só funciona em desktop.

## Acessibilidade

Preservar:

- contraste adequado
- texto legível
- tamanho confortável
- foco visível quando aplicável
- hierarquia compreensível
- semântica HTML quando possível

Não sacrificar legibilidade por sofisticação visual.

## Referências de rigor

Usar como referência de maturidade:

- Linear
- Stripe
- Vercel
- Figma
- Notion
- IBM Carbon
- Atlassian Design System
- Adobe Spectrum

Não copiar estética diretamente.

Usar como referência de rigor, consistência e maturidade sistêmica.
