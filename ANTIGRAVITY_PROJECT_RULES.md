## 1. Objetivo do projeto

Este portfolio não deve parecer um portfolio visual genérico.

Ele deve posicionar Henrico Amaral como Product Designer sênior especializado em:
- sistemas complexos
- plataformas operacionais
- ambientes enterprise
- operações críticas
- visualização de decisão
- estruturação de fluxos e informação

A estética deve ser editorial, técnica, sóbria e precisa.

## 2. Narrativa da home

A home deve funcionar como narrativa progressiva:

1. Hero
   Apresenta posicionamento e promessa central.

2. Método
   Explica a forma de pensar e estruturar problemas.

3. Cases
   Mostra evidências concretas do repertório.

4. Impacto
   Conecta design com clareza operacional e decisão.

5. Sobre
   Humaniza sem cair em autobiografia genérica.

6. Contato
   Fecha com autoridade e clareza.

Cada seção deve avançar a narrativa. Nenhuma seção deve parecer decorativa.

## 3. Regras visuais

Preservar:
- estética escura editorial
- grids sutis
- motion discreto
- atmosfera técnica
- textos grandes com boa respiração
- microtextos editoriais
- contraste entre massa escura e acentos precisos

Evitar:
- visual SaaS genérico
- excesso de cards
- excesso de blur
- efeitos chamativos
- animações rápidas
- visual futurista artificial
- linguagem corporativa vazia

## 4. Backgrounds e motion

Os backgrounds devem ter coerência entre seções.

A home deve parecer um sistema visual contínuo, não um conjunto de blocos isolados.

Motion deve ser:
- sutil
- lento
- elegante
- quase imperceptível
- subordinado ao conteúdo

Nunca usar motion que dispute atenção com texto.

## 5. Texto e copy

O texto deve ser:
- claro
- humano
- estratégico
- sem jargão vazio
- sem tom acadêmico
- sem autoelogio exagerado
- sem frases motivacionais

O tom deve comunicar senioridade por precisão, não por grandiosidade.

Evitar:
- transformar complexidade em clareza
- soluções inovadoras
- experiências incríveis
- design centrado no usuário como clichê solto
- impacto estratégico sem evidência

Preferir:
- decisão
- sistema
- operação
- contexto
- trade-off
- clareza
- estrutura
- consequência

## 6. Cases

Os cases principais são:

- Petrobras / SALACAR
- Bayer
- Ambev
- BMG

Eles devem ser tratados como uma trilogia expandida de sistemas complexos:
- Petrobras: operação crítica offshore
- Bayer: interpretação e validação de dados agrícolas
- Ambev: monitoramento logístico em escala
- BMG: jornada bancária PJ

A seção de cases deve vender repertório, não apenas imagem bonita.

## 7. Imagens

As imagens dos cases estão em WebP.

Antes de trocar qualquer imagem, verificar:
- caminho no HTML
- caminho no CSS
- peso do arquivo
- proporção
- consistência estética
- fallback se necessário

Não deixar referências quebradas para .png se o arquivo agora é .webp.

## 8. Design system

Arquivos de design system ficam em:

docs/

Não deixar arquivos soltos na raiz se forem documentação.

Qualquer link para design-system.html deve apontar corretamente para:

docs/design-system.html

ou para o arquivo correto dentro de docs.

## 9. Modo de trabalho obrigatório

Antes de alterar:
1. git status
2. confirmar branch
3. listar arquivos afetados
4. explicar risco
5. propor plano curto

Durante alteração:
- mexer em poucos arquivos por vez
- não refatorar junto com mudança visual pesada
- não alterar copy e layout ao mesmo tempo sem necessidade

Depois de alterar:
1. indicar o que mudou
2. indicar como testar
3. sugerir commit
4. apontar pendências

## 10. Critério de aceite

Uma alteração só é boa se:
- melhora a narrativa
- reduz inconsistência
- preserva senioridade
- não quebra responsividade
- não quebra links
- não aumenta ruído visual
- não cria dívida técnica desnecessária
